1.存储引擎

Mysql体系结构

连接层

最上层是一些客户端和链接服务，包含本地sock 通信和大多数基于客户端/服务端工具实现的类似于 TCP/IP的通信。主要完成一些类似于连接处理、授权认证、及相关的安全方案。在该层上引入了线程 池的概念，为通过认证安全接入的客户端提供线程。同样在该层上可以实现基于SSL的安全链接。服务 器也会为安全接入的每个客户端验证它所具有的操作权限。

服务层

第二层架构主要完成大多数的核心服务功能，如SQL接口，并完成缓存的查询，SQL的分析和优化，部 分内置函数的执行。所有跨存储引擎的功能也在这一层实现，如 过程、函数等。在该层，服务器会解 析查询并创建相应的内部解析树，并对其完成相应的优化如确定表的查询的顺序，是否利用索引等， 最后生成相应的执行操作。如果是select语句，服务器还会查询内部的缓存，如果缓存空间足够大， 这样在解决大量读操作的环境中能够很好的提升系统的性能。

引擎层

数据存储层， 主要是将数据(如: redolog、undolog、数据、索引、二进制日志、错误日志、查询 日志、慢查询日志等)存储在文件系统之上，并完成与存储引擎的交互。 和其他数据库相比，MySQL有点与众不同，它的架构可以在多种不同场景中应用并发挥良好作用。主要 体现在存储引擎上，插件式的存储引擎架构，将查询处理和其他的系统任务以及数据的存储提取分离。 这种架构可以根据业务的需求和实际需要选择合适的存储引擎。

存储引擎分类

InnoDB

Mysql5.5之后的默认存储引擎。

1)特点：

  • DML遵循ACID模型，支持事务！！！！！
  • 行级锁，提高并发性能
  • 支持外键约束，可以保证数据完整性

2)逻辑存储结构

  1. 表空间：InnoDB存储引擎逻辑结构的最高层，在表空间中可以 包含多个Segment段。
  2. 段：表空间由各个段组成，在InnoDB中由系统管理段，不需要人为控制，一个段包含多个区
  3. 区：区是表空间的单元结构，每个区大小为1M，一个区中有64个连续的页
  4. 页：页是组成区的单元，每个页大小默认是16KB，是InnoDB 存储引擎磁盘管理的最小单元！！！！
  5. 行：在InnoDB中存储是存储在行的。

3)适用场景

对事务的完整性有比较高的要求，在并发条件下要求数据的一致性，数据操作除了插入和查询之外，还包含很多的更新、删除操作

MyISAM

Mysql5.5之前的默认存储引擎。

1)特点：

不支持事务

不支持外键 支持表锁

不支持行锁 访问速度快

2)适用场景

以读操作和插入操作为主，少量更新和删除，对事务、并发没有要求

Memory

Mysql数据存储在内存中的存储引擎。

1)特点：

hash索引

2)适用场景

数据保存在内存，访问速度快，通常用作缓存。无法保证数据安全，存储上限低

小结

InnoDB和MyISAM的引擎区别？

  1. InnoDB支持事务，MyISAM不支持
  2. innoDB支持行锁表锁，MyISAM仅支持表锁
  3. InnoDB支持外键，MyISAM不支持

2.索引

简介

索引是mysql中帮助高效获取数据的有序数据结构，mysql在数据之外还维护着使用特定算法的数据结构，这些数据结构指向存储的数据，这样就可以实现快速查找，这种数据结构就是索引

特点

  1. 优势：
    • 可以提高数据查询速度降低IO成本
    • 通过索引查询有效降低cpu排序成本
  2. 劣势：
    • 索引列也会占用存储空间，庞大索引列反而有可能降低效率
    • 索引提高了查询速度，但是增加了插入更新操作的成本，效率降低

索引结构

mysql中支持很多索引类型，我们只需要了解InnoDB的B+树索引

B树

要了解B+树首先需要明确B树的概念：B树是一种多路平衡查找树，B树相对于二叉树可以有多个分叉。

下图以一颗度数为5的B树为例，则每个节点最多存储4个key，5个指针。

一旦节点存储的key数量达到5，就会裂变，中间元素向上分裂。

在B树中，叶子节点和非叶子节点都可以存放数据

B+树

B+树是B树的变种，绿色部分即为索引部分不存储数据，红色部分存储数据。存储数据的叶子节点部分形成一个单向链表。

Mysql中索引使用的并不是标准的B+树：

Mysql在B+树的基础上增加了指向相邻叶子节点的链表指针，形成了顺序的B+树，适应mysql中的排序场景。

小结

为什么么InnoDB存储引擎选择使用B+树索引结构？

  • 相对于二叉树，层级少，搜索效率高；
  • B树中无论叶子节点还是非叶子节点都会保存数据，想保存大量数据，就会增加树的高度，导致性能降低

索引分类

分类 含义 特点 关键字
主键索引 针对于表中主键创建的索引 只能有一个，默认自动创建 PRIMARY
唯一索引 避免同一个表中某数据列中的值重复 可以有多个 UNIQUE
全文索引 快速定位特定数据 可以有多个
常规索引 全文索引查找的是文本中的关键词，而不是比 较索引中的值 可以有多个 FULLTEXT

聚集索引、二级索引

分类 含义 特点
聚集索引 将数据存储与索引放到了一块，索引结构的叶子 节点保存了行数据 必须有,而且只 有一个
二级索引 将数据与索引分开存储，索引结构的叶子节点关 联的是对应的主键 可以存在多个

如果存在主键索引，主键索引就是聚集索引，如果不存在则选择第一个UNION索引作为聚集索引，如果没有主键没有唯一索引，mysql会自动创建一个隐藏的rowid作为索引

  • 聚集索引中叶子节点挂着这一行的数据
  • 二级索引中叶子节点挂着该字段的主键值

查找过程：

由图可知根据name字段查询，先根据name=‘Arm’找到二级索引，但是二级索引中只存储着对应的主键值10，由于需要返回的字段是*所以还需要根据主键10在聚集索引中查询对应的行。最终拿到这条记录返回。这种先到二级索引中查找数据，找到主键再根据主键在聚集索引中获取数据的方式称之为回表查询！！！！！日常使用中要尽量避免回表查询。

索引使用

1.最左前缀法则

​ 如果索引了多列（联合索引），要遵守最左前缀法则。最左前缀法则指的是查询从索引的最左列开始， 并且不跳过索引中的列。如果跳跃某一列，索引将会部分失效(后面的字段索引失效)，查询时，最左变的列必须存在，否则索引全部失效。

2.范围查询

​ 出现范围查询(大于小于)，范围查询右侧的列索引失效。在大于等于（>=）的情况下索引生效。

3.索引失效情况

  • 索引列上进行计算，索引将失效。
  • 字符串不加引号，索引将失效
  • 模糊查询如果是头部模糊匹配 ，索引将失效
  • or连接条件前的列有索引，后面的列没有索引，索引将失效
  • 如果mysql觉得使用索引比全表查询慢不会使用索引

4.sql提示

  1. use index 建议使用的索引，只是建议mysql内部会自己评估
  2. ignore index 忽略指定索引
  3. force index 强制使用索引

5.覆盖索引

覆盖索引指的是查询使用索引，且需要返回的列，在索引中可以全部找到（覆盖）

使用explain查询sql执行计划，表格最后一个字段Extra会根据使用索引的情况分为以下两种情况：

Extra 含义
Using where; Using Index 查找使用了索引，但是需要的数据都在索引列中能找到，所以不需 要回表查询数据
Using index condition 查找使用了索引，但是需要回表查询数据

尽量避免回表查询！！！

前缀索引

​ 当字段类型为text、varchart等时，索引需要很大的字符串，查询慢，可以将一部分前缀建立索引，大大节约空间。

索引设计原则

1). 针对于数据量较大，且查询比较频繁的表建立索引。

2). 针对于常作为查询条件（where）、排序（order by）、分组（group by）操作的字段建立索 引。

3). 尽量选择区分度高的列作为索引，尽量建立唯一索引，区分度越高，使用索引的效率越高。

4). 如果是字符串类型的字段，字段的长度较长，可以针对于字段的特点，建立前缀索引。

5). 尽量使用联合索引，减少单列索引，查询时，联合索引很多时候可以覆盖索引，节省存储空间， 避免回表，提高查询效率。

6). 要控制索引的数量，索引并不是多多益善，索引越多，维护索引结构的代价也就越大，会影响增 删改的效率。 1 create unique index idx_user_phone_name on tb_user(phone,name);

7). 如果索引列不能存储NULL值，请在创建表时使用NOT NULL约束它。当优化器知道每列是否包含 NULL值时，它可以更好地确定哪个索引最有效地用于查询。

3.Sql优化

插入数据优化

  1. 批量插入数据
  2. 手动控制事务
  3. 主键顺序插入

主键优化

在InnoDB中表数据都是根据主键顺序组织存放的，行数据都是存储在聚集索引的叶子节点上的，在InnoDB引擎中数据行是在页中存放的，每个页固定大小16k，如果存储过大将会放到下一页，页与页通过指针相连。

主键顺序插入性能是要高于乱序插入的。原因是乱序插入会造成页分裂

页分裂：

假设有mysql中有两页存顺序放着正好填满两页大小的数据，页与页直接用指针相连。此时插入一个中间值，不会再次申请一页来存放数据，因为索引结构中的叶子节点是有顺序的。mysql会将插入值应在页的后半页移动到新的页然后再新页插入数据，这样做完之后页与页的顺序是乱的，所以此时需要重新设置链表指针。这种现象就被称为页分裂。

页合并：

当删除一行记录时，实际上并没有被删除，只是被标记为删除且它的存储空间允许别的记录声明使用。当页中的数据删除记录达到页的50%，InnoDB会开始寻找靠近的页，看看是否可以将两个页合并使用。这种现象叫页合并。

order by优化

mySql中有两种排序方式：

Using filesort：通过表的全表扫描或索引获取数据，在排序缓冲区中进行排序操作。所有不是通过索引直接返回排序结果的排序都叫 FileSort 排序。

Using index：通过有序索引顺序扫描直接返回有序数据，这种情况即为 using index，操作效率高。尽量使用Using index！！！！

优化原则：
  1. 根据排序字段建立索引遵循最左前缀法则。
  2. 使用覆盖索引。
  3. 多字段排序尽量针对每个字段创建升序或降序的索引。

groupBy优化

优化原则：
  1. 建立索引遵循最左前缀法则。

count优化

优化原则：
  1. 建立索引遵循最左前缀法则。
  2. count(字段) < count(主键 id) < count(1) ≈ count(*)，所以尽 量使用 count(1)

update优化

当执行update语句时，会锁定这一行的数据，事务提交之后释放锁，在开启多个事务执行时，会将行锁升级为表锁，导致性能降低。

优化原则：
  1. InnoDB的行锁是针对索引加的锁，不是针对记录加的锁 ,并且该索引不能失效，否则会从行锁 升级为表锁

4.Mysql中的锁

分类

  1. 全局锁：锁定数据库中的所有表。
  2. 表级锁：每次操作锁住整张表。
  3. 行级锁：每次操作锁住对应的行数据。

全局锁

全局锁就是针对整个数据库加锁，整个实例会处于只读状态。

表级锁

表级锁，每次操作锁整张表。主要分为表锁、元数据锁、意向锁。

表锁

  • 读锁 不会阻止其他客户端的读，但是会组织写
  • 写锁 写锁即会组织读也会阻止写

元数据锁

元数据锁为mysql自动添加无需手动控制。

意向锁

使用表必须检查每一行是否加锁，需要从第一行检查到最后一行效率低下，mysql引入了意向锁来判断是否加锁:lock:在加行锁的时候会添加意向锁，另一个线程加锁时只需判断意向锁:lock:

分类：

  1. 意向共享锁
  2. 意向排他锁

行级锁

行锁通过索引的索引项来加锁，而不是对记录加锁。:lock:

  1. 行锁 锁定单个行记录的锁，RC、RR隔离级别都支持
  2. 间隙锁 锁定索引间隙的锁，防止插入操作产生幻读。RR级别支持
  3. 临键锁 行锁和间隙锁的组合，同时锁定数据与间隙，RR隔离级别支持
